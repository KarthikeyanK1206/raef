RAEF — Reliable Agent Execution Framework

A log-first runtime that makes LLM-agent tool execution durable across crashes, restarts, and ambiguous network failures — every side effect journaled before it is dispatched, replayed safely after failure.

Section 1

Overview

An agent that books a flight or writes to a database has a side effect it cannot take back. If the process dies after the request is sent but before the response is recorded, the naive options are both wrong: retry and you double-book; skip and you lose the booking. Worse, a timeout gives you no information at all — the write may or may not have happened.

RAEF sits between the agent loop and its tools and solves this with four mechanisms:

  1. Log before send. The tool intent — name, arguments, deterministic execution ID — is committed to SQLite before the network call starts. A crash at any point leaves a durable record of what was in flight.
  2. Deterministic identity. execution_id = sha256(run_id | plan_item_id | tool_name | canonical_json(args)). A restarted process recomputes the same ID from the same logical call and finds the durable record — this is what makes replay reuse instead of re-execute.
  3. Explicit ambiguity. A write whose outcome is unknown raises AmbiguousToolError and is parked as pending_recovery. It is never silently retried.
  4. Verification-driven recovery. A pluggable verifier probes the target system: proven commit → mark verified_committed and reuse the result; proven absence → mark verified_not_found and replay exactly once; inconclusive → hand off to a human or policy.

Elevator pitch. "I built a crash-durable execution runtime for LLM agents: a write-ahead log over SQLite with deterministic tool-execution IDs and an explicit six-state execution lifecycle. Kill the process at any point — including mid-write — and the restarted agent replays to where it stopped with exactly one commit landing at the target, which the demos verify from the target's own commit counter."

Everything runs locally with zero runtime dependencies: persistence is stdlib sqlite3, hashing is hashlib, serialization is json. A mock target service and a deterministic crash injector make every failure mode reproducible on a laptop and in CI — no API keys, no network.

The system is layered like a small database: an append-only event journal is the source of truth, mutable projections make reads cheap, and a facade guarantees the two never disagree.

The execution state machine

Every tool call moves through an explicit lifecycle, and the facade rejects illegal transitions — orchestration bugs cannot, for example, move a verified-committed write back to dispatched.

intent_logged→ dispatched→ acked happy path: durable intent, send, durable result
dispatched→ timeout / crash (ambiguous)→ verify against target…
↳verified_committed proof of commit → reuse the durable result, never re-send
↳verified_not_found→ acked proof of absence → replay exactly once (idempotency key = execution ID)
↳failed definitive error → surfaced to policy/human; retriable via explicit transition only

Terminal, reusable states: acked verified_committed. Recovery-eligible states: intent_logged dispatched verified_not_found.

Component by component

ModuleRoleThe detail worth remembering
logging_service.py Facade over planner, context, results, and WAL. Owns run lifecycle, checkpoints, soft JSON mirrors. Every public method takes a per-run re-entrant lock and opens one BEGIN IMMEDIATE SQLite transaction, so events and projections commit atomically. Event payloads are schema-validated per event type before append.
txn_manager.py Log-before-send tool execution. Builds execution IDs, dedups, classifies outcomes. On replay it distinguishes: terminal success → REUSED; failed → surfaced, never auto-retried; ambiguous → PENDING_RECOVERY; verified_not_found → the one state allowed to re-invoke the tool.
runtime_store.py Single canonical SQLite layer: 9 tables (planner, messages, results, payload artifacts, WAL events, checkpoints, evaluation spans). journal_mode=WAL + synchronous=FULL = crash durability at fsync cost. Payloads > 16 KB move to a content-addressed artifact table (SHA-256) instead of inflating hot rows.
wal.py Backend-agnostic WAL protocol + SQLite implementation. Checkpoints must reference an existing event seq_id in the same run — enforced at write time, audited by the CLI.
recovery/ Post-restart coordinator, decision strategy, verifier protocol, health checkers, pulse monitor. Strategy maps state → decision (resume_waiting, replay, mark_committed, handoff). With a verifier wired, ambiguous writes are resolved by probing the target instead of handing off.
verifier.py Commit probe for the mock target; WriteVerifierProtocol is the contract for real targets. Two proofs: execution ID present in the target's action log, or domain state already matching the intended payload.
evaluation.py Durable timing spans (step attempts, phases) stored in the same SQLite file. Spans are opened before work starts, so a crash leaves an open span that reports as interrupted — crash visibility comes free from durability.
cli.py Operator CLI: runs · show · events · executions · pending · report · recover · audit. audit checks log-first invariants (dangling pending IDs, intents without result records, checkpoint ahead of log) and exits non-zero on violation.
tools/ Mock agent, mock target service, crash injector. The mock target supports four API levels (legacy, idempotent, RIFL-style probeable, queryable-distinguishable) — the framework's guarantees can be tested against each contract strength.
adapters/ Decorators (with_logging_service, ensure_run_started) and a LangChain-style tool wrapper with policy gates. ToolPolicy carries read-only / concurrency-safe / requires-permission metadata; permission denial raises before any intent is logged.

Section 4

Lifecycle walkthrough: one booked flight, three failure stories

The demo task books flight RA512 against the mock target. The same logical write is driven through three injected failures; in every case the run ends with the order booked and the target's own commit counter at exactly 1.

A — Hard crash after the write committed

  1. Intent logged → dispatched → target commits → result acked (all durable).
  2. SimulatedCrash kills the process (exit 42) right after the tool transaction; the step's timing span is left open.
  3. Restart with the same run_id: the agent replays the script, recomputes the same execution ID, finds the acked record → disposition REUSED. The tool is not re-invoked. The open span reports as interrupted; attempt 2 succeeds.

B — Ambiguous write, response lost

  1. The target commits, but the adapter raises AmbiguousToolError before the ack is observed — the classic "timeout after send".
  2. The record is parked timeout / dispatched and stays in the pending set. Crash, restart.
  3. On resume the transaction manager returns PENDING_RECOVERY; the coordinator waits out the configured window, then asks the verifier. The execution ID is found in the target's action log → verified_committed, the durable result is reconciled into the transcript as a tool message, pending is cleared. No second commit.

C — Ambiguous write, request lost

  1. The adapter raises before the request ever reaches the target — same client-side symptom as B, opposite truth.
  2. After restart, verification finds no record and the domain state unchanged → verified_not_found, decision REPLAY.
  3. The transaction manager re-invokes the tool — the one state where re-execution is permitted — passing the execution ID as an idempotency key, and the write lands once: acked.

The line that wins the interview: B and C are indistinguishable from the client's point of view at failure time. The only correct tiebreaker is evidence from the target, which is why recovery is verification-driven rather than retry-driven.

Section 6

Key design decisions

Event journal + mutable projections, one transaction

Decision Append validated WAL events and update the planner/context/result projections inside the same SQLite transaction.
Alternative Pure event sourcing (rebuild state by folding events) or projections only (no journal).
Why Reads stay O(1) without replaying the log; the journal remains the audit trail and recovery source; atomic commit means the two can never disagree — which pure "write log, then update state" designs must handle as a partial-failure case.

Deterministic execution IDs instead of stored UUIDs

Decision Hash the logical call identity (run_id | plan_item_id | tool_name | canonical args, optional invocation key) into the execution ID.
Alternative Generate a UUID at first attempt and persist it.
Why A restarted process may crash before a UUID could be persisted-and-associated; a deterministic ID is recomputable from the replayed call itself, so dedup works even for the earliest crash windows. The invocation_key escape hatch handles intentional repeat calls with identical args.

SQLite with synchronous=FULL as the only store

Decision One local SQLite file in WAL mode, fsync on every commit; JSON files are demoted to non-canonical mirrors.
Alternative Postgres/Redis service, or JSON files as the store.
Why The unit of failure being defended is the process, and SQLite gives real transactional durability with zero operational surface — right-sized and honest for a local-first framework. The WAL interface is a protocol, so a server-backed store is a swap, not a rewrite. Cost: ~fsync per event batch (visible in the timing report), single-writer-per-file scaling.

Ambiguity is a first-class state, not an exception path

Decision Timeouts on writes park the execution as pending; only target-side evidence (or an explicit human decision) moves it on. Illegal transitions raise.
Alternative Retry-with-backoff by default, as most HTTP clients do.
Why Blind retry is exactly the double-booking bug this framework exists to prevent; for non-idempotent side effects the only safe default is to stop and verify. The enforced state machine makes "safe to replay" a provable property of a record rather than a comment.

Verify-then-replay leans on target idempotency — stated, not hidden

Decision Replay after verified_not_found passes the execution ID as an idempotency key; the docs state plainly that against a target with no dedup there remains a window (verify says absent; the original late request lands after the replay) no client alone can close.
Why That residual race is a fundamental distributed-systems limit (it is the problem RIFL-style request layers solve server-side). Claiming "exactly-once" unqualified would be false; "effectively-once given an idempotent target, at-most-once dispatch per proven state otherwise" is defensible and demonstrated.

Failure injection as deterministic phases, not chaos

Decision The demo threads named crash phases (before_tool_transaction, after_tool_transaction, …) through the step loop; the e2e driver runs the agent as a child process and asserts on exit code 42, then on the restarted run.
Alternative Random kill-based chaos testing.
Why Each crash point maps to a specific recovery claim, so every claim has a reproducible test — and the same demos run in CI because they need no network.

Section 9

Strengths, limitations, future work

Strengths to lead with

  • Correctness core: log-before-send + deterministic IDs + enforced state machine.
  • Both ambiguity directions resolved by evidence, verified from the target's own counters.
  • Zero-dependency stdlib core; everything reproducible offline, including in CI.
  • Failure injection is deterministic — every recovery claim has a named, repeatable test.
  • Operability: an audit command that turns invariants into exit codes; durable timing that classifies crashes as interrupted attempts.

Limitations (say them before they're asked)

  • Single-process, single-writer per run; durability is one machine's disk — no replication or HA.
  • Verify-then-replay is fully safe only when the target dedupes by execution ID; the residual late-duplicate window is documented, not solved.
  • WAL grows unboundedly — checkpoints exist but log truncation/compaction is not implemented.
  • Recovery verification is wired for the mock target; real connectors (HTTP idempotency keys, payment-style check APIs) are contracts, not implementations.
  • Payloads are stored in plaintext; no redaction, encryption, or access control.

Future work, in priority order: WAL truncation after checkpoint (bounded storage); a Postgres store behind the existing WAL protocol (multi-worker, row-level run locks); recovery-time budget with escalation policies; payload redaction hooks; a verifier implementation for a real HTTP service with idempotency keys.

Section 11

Interview questions & answer outlines

Click a question to self-test before revealing the outline.

Architecture

Why log the intent before dispatching the tool call?
If you send first and log after, a crash in between leaves an untracked side effect — the worst failure mode, invisible on restart. Logging first means the recovery scan always has a record of anything possibly in flight; the cost is a benign "intent logged, never sent" record, which recovery classifies as safe to replay (for writes, the dispatch marker is written before invoke, so intent_logged means nothing was sent).
Walk me through what happens when the process dies mid-write.
Durable state: intent + dispatched marker. On restart: same inputs → same execution ID → record found in dispatched. Transaction manager returns PENDING_RECOVERY without touching the target. Coordinator waits out the in-flight window, then verifier probes the target: found → verified_committed, result reconciled into the transcript; absent → verified_not_found → single safe replay with idempotency key; probe fails → handoff. Every transition is validated against the allowed-transitions table.
Why both an event log and mutable projections? Isn't that redundant?
They answer different questions: projections answer "what is the state now" in O(1); the journal answers "how did we get here" and survives projection bugs. Written in one SQLite transaction they can't diverge. Pure event sourcing makes every read a fold; projections-only loses the audit trail and recovery ordering. Checkpoints tie a snapshot to a log position (seq_id) so future truncation is possible.
How is the execution ID built, and what about collisions?
SHA-256 over run_id | plan_item_id | tool_name | canonical JSON args (sorted keys), truncated to 96 bits. Identical logical calls colliding is the feature (dedup); accidental collisions at 96 bits are negligible at agent scale. Intentional repeats with identical args are disambiguated with an explicit invocation_key. Cross-run reuse of an ID is rejected at the facade.
Why a facade (LoggingService) instead of letting components write to the store directly?
Single choke point for the three invariants: per-run mutual exclusion (RLock), atomicity (one transaction per operation), and legality (event payload validation + state-machine enforcement). Components stay composable — txn manager, recovery, evaluation don't know about each other — and the store can be swapped behind the WAL protocol.

Tradeoffs

Why SQLite and not Postgres or Redis?
The failure unit defended here is the process, not the machine fleet. SQLite gives real ACID + WAL durability with zero ops, works offline, and makes the demos/CI reproducible. Redis (default config) can lose acknowledged writes on crash — wrong tool for a durability core. Postgres is the right growth path and the WAL protocol is designed as the seam; what changes is locking (per-run advisory/row locks) not the model.
synchronous=FULL makes every commit fsync. Defend that.
The entire value proposition is "acked means durable"; NORMAL in WAL mode can lose the last commits on OS/power failure. ~15 ms per step is dominated by fsync and that's the honest price of the guarantee. Mitigations if needed: batch events per step into one commit (group commit), or offer a documented relaxed mode for non-critical runs — a policy knob, not a redesign.
What does checkpoint frequency trade off?
Checkpoints snapshot projections at a log position. More frequent → faster recovery reads and earlier truncation potential, but more write amplification per event. It's configurable (checkpoint_every_n_events, demos use 4). Since projections are also durable, checkpoints here mainly bound the "events after checkpoint" replay window and enable future log compaction.
Why enforce the state machine in code rather than documenting it?
Recovery decisions key off states; a corrupted state (e.g. verified_committed → dispatched) could re-fire a committed write — the exact catastrophe the system prevents. Raising on illegal transitions turns orchestration bugs into loud failures. It also forced a real design decision: replay after verified_not_found deliberately skips the dispatch marker because that transition is illegal, and repeat ambiguity during replay stays in verified_not_found for re-verification.

Scalability

How would this scale to many concurrent agents or workers?
Today: one process, per-run locks, SQLite single-writer — fine for its scope. Scaling path: keep the model, swap the store — Postgres behind the WAL protocol, per-run advisory locks or SELECT … FOR UPDATE for mutual exclusion, workers claiming runs. The recovery scan is already run-scoped, so sharding by run_id is natural. The deterministic-ID dedup needs no change — it's storage-agnostic.
The WAL grows forever. What's your plan?
Named limitation. Checkpoints already bind snapshots to seq_id, so truncation = delete events ≤ last checkpoint that aren't needed for open executions, keeping terminal external-result records (they're the dedup source). I'd implement it as a raef compact CLI verb with an audit run before and after.
What happens with very large tool payloads?
Payloads over 16 KB inline limit move to a content-addressed artifact table (SHA-256 keyed), keeping hot rows and checkpoint snapshots small; records carry a typed ref (id, size, hash). Next step for truly large blobs is spilling artifacts to files/object storage behind the same ref shape.

Failure modes

Two writes time out identically — one committed, one never arrived. How do you tell them apart?
You can't, from the client alone — that's the core insight. Both are timeout/dispatched locally. The tiebreaker must be target-side evidence: probe by execution ID (or domain readback). Found → reuse; provably absent → replay once; can't tell → do nothing automatic, hand off. This is why recovery is verification-driven, not retry-driven.
Is verify-then-replay actually safe? What's the race?
The race: verify says absent, you replay, then the original late request finally lands → duplicate. A client alone cannot close this window. It's closed by the target deduping on the idempotency key (the execution ID travels with both attempts — same key, one commit), which RIFL-style request layers formalize. Against a target with no dedup, RAEF's honest posture is: verified replay is a policy choice with a documented residual risk, default is handoff.
What if the crash happens during recovery itself?
Recovery is idempotent by construction: decisions are pure functions of durable records; marking committed re-checks the transcript before reconciling (no duplicate tool messages); replay goes through the same dedup'd transaction path. Run recovery twice and the second pass finds terminal states and does nothing.
What failure modes are NOT handled?
Disk loss/corruption of the SQLite file (single copy — needs backup/replication); Byzantine or lying targets (verification trusts the target's answer); concurrent writers on the same run from different processes (locks are in-process; the DB transaction still serializes but the design assumes one runner per run); clock skew only affects wait-window pacing, not correctness.
Why does a failed execution not auto-retry?
failed means the target answered definitively — retry is a policy question (was it transient? is the action still wanted?), not a runtime one. The state machine allows failed → dispatched so a policy layer can retry explicitly; the runtime just refuses to guess. Ambiguity (timeout) is deliberately a different state from failure.

Testing

How do you test crash recovery deterministically?
Two levels. Unit: fault-injecting adapters raise AmbiguousToolError before/after really mutating a real mock target, then assertions run on both RAEF state and the target's own counters. E2E: the driver spawns the agent as a child process, a crash simulator raises at a named phase (nine injection points), driver asserts exit code 42, restarts with the same run_id, and asserts the final state — same flow CI runs, no network.
What's the most important assertion in the suite?
target.store.stats["total_commits"] == 1 after each ambiguity scenario — it checks the business outcome (no double booking) from the target's own bookkeeping rather than trusting the framework's account of itself. Plus its mirror: replayed tools are invoked the expected number of times (1 for reuse, 2 for lost-then-replayed).
How did you test the audit tool?
Both directions: a clean seeded run exits 0 with "invariants hold"; then the test deliberately corrupts an invariant (adds a pending execution ID with no durable record) and asserts exit code 1 naming the violation. An auditor whose failure path is untested is decoration.

Observability

How do you observe a run in production terms?
Three layers: (1) durable evaluation spans — opened before work, so crashes are visible as interrupted attempts with per-phase timing; (2) structured stdlib logging at state transitions (WARNING on ambiguity, INFO on recovery decisions), handler-free as a library should be; (3) the CLI — events for the journal, executions for statuses, report for timing, audit for invariants as an exit code you can alert on.
Why store timing spans in the same SQLite file as the runtime?
Crash-consistency of the telemetry itself: a span opened before work and never closed is durable evidence of an interrupted attempt — an external metrics store would just show a gap. One transaction domain also means timing can't outlive or predate the state it describes. At scale you'd export from this file, not bypass it.

Security

What's the security posture?
Honest scope: local developer runtime — no auth, no encryption at rest, payloads stored plaintext (flagged limitation; redaction hooks are future work). What exists by design: ToolPolicy permission gates that deny before any intent is logged, read/write operation typing so policy can treat writes as privileged, and a full audit trail of every attempted side effect — which is the forensic property security teams actually want from agents.
Could a malicious tool result corrupt the runtime?
Results are treated as opaque JSON — validated for shape (dict), size-gated into artifact storage, never executed or templated. The state machine is driven by the runtime's own observations, not payload content. The realistic risk is prompt-injection at the agent layer, upstream of RAEF; RAEF's contribution there is the permission gate and the journal.

Performance

Where does the time go in a step?
From the durable report: ~15 ms per attempt; tool_transaction ~12.7 ms dominates, recovery ~5.5 ms, mock LLM ~1.8 ms. Within the transaction it's fsync cost — several durable commits (intent, dispatch, result, context) at synchronous=FULL. Real deployments invert the ratio: LLM and network latency dwarf journaling overhead, which is exactly the argument for paying it.
How would you cut the journaling overhead?
Group commit: batch the events of one logical operation into one transaction (already the facade's shape — intent+dispatch could merge); tune checkpoint interval; optionally expose a documented relaxed-durability mode. Measure first via the built-in phase report — the instrumentation to justify the optimization already exists.

Judgment & reflection

Why not just use LangGraph's checkpointing?
Different layer. Graph checkpointers persist conversation/graph state between steps; they don't give tool side effects an identity, a state machine, or ambiguity resolution — resume after a mid-write crash and the framework happily re-runs the node. RAEF targets the tool boundary specifically and is designed to sit inside such frameworks (the LangChain-style wrapper demonstrates the integration point).
What was the hardest design bug you hit?
Making replay legal. First cut re-dispatched verified_not_found executions — which the transition table correctly rejected (verified_not_found → dispatched is illegal, dispatch markers mean "first send"). The fix reshaped the feature: replay skips the dispatch marker, and repeat-ambiguity during replay stays in verified_not_found for re-verification rather than regressing to dispatched. The state machine caught my own design error — the best argument for having it.
What would you build differently starting over?
(1) Design the verifier contract on day one instead of retrofitting — ambiguity without resolution is half a feature. (2) Async-first API for concurrent tool calls (the pending set already supports multiple in-flight IDs; the execution API doesn't). (3) Log compaction from the start, since checkpoints only pay off when the log can shrink. I'd keep: stdlib-only core, deterministic IDs, deterministic fault injection.
